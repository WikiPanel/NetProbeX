# NetProbeX

NetProbeX is a Go-based network connectivity and stability probe system for checking VPN-related server reachability and transport behavior before building or choosing a real VPN transport path.

It is intentionally not a VPN tunnel and does not implement bypass logic. This phase focuses on connectivity, stability, protocol behavior, and real traffic testing across TCP, UDP, HTTP, WebSocket, TLS, and streamed downloads.

## Binaries

- `server-agent`: runs on Ubuntu 22.04 / Ubuntu 24.04 and exposes protocol test listeners.
- `client-probe`: runs primarily on macOS for development, with builds for Windows and Linux clients.

## What It Tests

- DNS resolution and latency
- TCP connect success, timeout, refused, reset, and repeated checks
- TCP echo stability with periodic real payloads
- UDP echo packet loss and latency
- HTTP `/ping` and `/health`
- WebSocket `/ws` stability with periodic payloads
- TLS handshake timing, version, and cipher suite
- Download stability and speed using generated bytes, including stalls and interrupted transfers

## Build

Build the current platform:

```bash
go build ./cmd/server-agent
go build ./cmd/client-probe
```

Build release binaries into `dist/`:

```bash
scripts/build-all.sh
```

Outputs include:

- `netprobex-server-linux-amd64`

## Server Installation

On Ubuntu:

```bash
scripts/build-all.sh
sudo scripts/install-server.sh
sudo nano /opt/netprobex/server.json
sudo systemctl start netprobex-server
sudo systemctl status netprobex-server
```

The service runs:

```text
/opt/netprobex/server-agent --config /opt/netprobex/server.json
```

Low ports such as `443` require root or the relevant Linux capabilities. Basic tests on high ports do not require special privileges.

## Server Usage

```bash
./server-agent --config configs/server.example.json
```

Server endpoints:

- TCP echo on configured `tcp_ports`
- UDP echo on configured `udp_ports`
- HTTP `GET /ping`
- HTTP `GET /health`
- HTTP `GET /stats`
- HTTP `GET /download/1mb`
- HTTP `GET /download/10mb`
- HTTP `GET /download/50mb`
- HTTP `GET /download/custom?size_mb=NUMBER`
- WebSocket `/ws`
- Optional TLS listener on `tls_port`

The HTTP and WebSocket services may share one port if `websocket_port` equals `http_port`; otherwise, two HTTP servers are started.

Note: one process cannot bind raw TCP echo and TLS to the same port at the same time. If `tls_port` overlaps a TCP port, that port is reserved for TLS and skipped by the raw TCP echo listener.

## Client Usage On macOS

```bash
./client-probe --config configs/client.example.json
```

With flags:

```bash
./client-probe --target example.com --duration 60s --tcp 443,8443,2053 --udp 443,8443 --download 10mb
```

The client runs continuous stability tests for the configured duration. TCP stability, UDP echo, WebSocket stability, repeated TCP connects, repeated HTTP checks, and downloads run concurrently.

## Client Usage On Windows

Build or copy `dist/netprobex-client-windows-amd64.exe`, then run from PowerShell:

```powershell
.\netprobex-client-windows-amd64.exe --config configs\client.example.json
```

Or:

```powershell
.\netprobex-client-windows-amd64.exe --target example.com --duration 60s --tcp 443,8443 --udp 443
```

## Config Examples

Server:

```json
{
  "tcp_ports": [443, 8443, 2053, 2083, 2096],
  "udp_ports": [443, 8443, 2053],
  "http_port": 8080,
  "websocket_port": 8081,
  "tls_port": 9443,
  "tls_cert_file": "",
  "tls_key_file": "",
  "log_file": "/var/log/netprobex/server.jsonl"
}
```

Client:

```json
{
  "target": "example.com",
  "duration_seconds": 60,
  "timeout_seconds": 5,
  "tcp_ports": [443, 8443, 2053, 2083, 2096],
  "udp_ports": [443, 8443, 2053],
  "http_base_url": "http://example.com:8080",
  "websocket_url": "ws://example.com:8081/ws",
  "tls_address": "example.com:9443",
  "download_tests": ["/download/1mb", "/download/10mb"],
  "output_json": "netprobex-report.json",
  "allow_insecure_tls": true
}
```

If no certificate paths are supplied, the TLS listener generates an in-memory self-signed certificate for testing. Use `allow_insecure_tls: true` on the client for that mode.

## Logs

Server logs are JSONL, one JSON object per line:

```json
{"time":"2026-05-28T00:00:00Z","event":"tcp_connected","client_ip":"1.2.3.4","protocol":"tcp","port":443}
```

Common events:

- `tcp_connected`, `tcp_disconnected`, `tcp_read_error`
- `udp_packet`, `udp_idle_timeout`
- `download_started`, `download_completed`, `download_interrupted`
- `websocket_open`, `websocket_close`, `websocket_error`
- `tls_handshake`, `tls_handshake_error`

## Report

The client prints a human-readable terminal report and writes a structured JSON report to `output_json`.

The JSON report includes DNS, TCP connect, TCP stability, UDP, HTTP, WebSocket, TLS, download results, warnings, scoring, and the best candidate transport path.

### Scoring System

NetProbeX scores each measured area from `0` to `100` and labels it as `stable`, `mostly_stable`, `unstable`, `poor`, or `failed`.

Structured scores are written under `transport_scores`:

- `dns`: resolution success and resolution time.
- `tcp`: repeated connection success plus long-running TCP echo stability.
- `udp`: packet return rate, packet loss, timeout count, and latency stability.
- `http`: repeated `/ping` and `/health` success rate, timeout count, latency stability, and download contribution.
- `websocket`: sustained message exchange, disconnect count, reconnect count, failures, and latency stability.
- `tls`: handshake success, handshake time, TLS version, cipher suite, and whether a stable `wss://` WebSocket path is also available.
- `download`: completion percentage, full completion count, partial completion count, stalls, interruptions, and speed drops.

The final ranked candidate list uses selectable transports: WebSocket, HTTP, TLS, Raw TCP, and UDP. DNS is informational, and download stability contributes to HTTP because downloads are HTTP traffic.

Long-running stability is weighted higher than initial connectivity. A TLS handshake alone is capped because it proves that TLS can start, not that sustained traffic remains stable. A stable WebSocket session should rank above raw TCP when raw TCP has drops. UDP with 100% packet loss scores `0` and should not be selected as best.

Download labels are more specific:

- `completed`: all expected bytes were received.
- `partial`: most bytes were received but the transfer did not fully complete.
- `unstable/interrupted`: the transfer stalled, was cut off early, or failed.

Download results include `completion_percent`, `partial_completion`, `interrupted`, and `interruption_reason`. Reasons include `client_timeout`, `server_disconnect`, and `unknown_error` when the client can infer them.

Example JSON scoring block:

```json
{
  "transport_scores": {
    "websocket": {
      "score": 92,
      "status": "stable",
      "reason": "WebSocket exchanged 112/112 messages over 120s with 0 disconnects."
    },
    "udp": {
      "score": 0,
      "status": "failed",
      "reason": "No UDP echo replies were received."
    }
  },
  "best_candidate": {
    "transport": "websocket",
    "candidate": "WebSocket over TLS/443",
    "score": 92,
    "status": "stable",
    "reason": "Sustained message exchange ranked above initial connection success."
  }
}
```

## Troubleshooting

- Permission denied on port 443: run the server as root, use systemd, or choose high ports such as 8080/8443.
- TCP succeeds but stability fails: the route may accept initial connections but drop once real payloads begin.
- UDP has high loss: UDP may be filtered, shaped, or unstable on the path.
- WebSocket connects then drops: check intermediaries, reverse proxies, idle timeouts, and frame filtering.
- Download starts fast then slows sharply: the report flags speed drops after the initial burst.
- TLS fails with a self-signed certificate: set `allow_insecure_tls` to `true` for testing.
- TLS listener skipped: make sure `tls_port` is not already used by HTTP/WebSocket and that the certificate paths are readable if you configured them.

## Security Notes

NetProbeX is a measurement tool. Do not expose it broadly without firewall rules.

- Download endpoints intentionally generate bandwidth.
- Logs include client IPs and user agents.
- Self-signed TLS is for testing only.
- No packet capture is implemented in this phase.
- No VPN tunneling or censorship bypass is implemented in this phase.
